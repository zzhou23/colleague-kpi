### 工作流程

```
员工日常使用 Claude Code
         ↓

         ↓

         ↓

         ↓
```

## 数据采集范围

我们读取 `~/.claude` 目录下的所有有价值数据：

| 数据源 | 文件/目录 | 能挖出什么 |
|--------|-----------|-----------|
| 命令历史 | `history.jsonl` | 每一条输入的内容、时间戳、所属项目、会话 ID |
| 会话记录 | `sessions/*.json` | 对话轮数、使用了哪些工具、上下文压缩次数、会话时长 |
| 项目记忆 | `projects/*/memory/` | 是否主动积累知识、记忆文件数量和更新频率 |
| 配置文件 | `settings.json` | 模型选择偏好、权限配置、Hook 自动化设置 |
| 个人规则 | `rules/`、`CLAUDE.md` | 是否建立了个性化开发规范 |
| 任务管理 | `tasks/` | 任务创建数、完成数、完成率 |
| 规划记录 | `plans/` | 是否使用 AI 的规划能力处理复杂任务 |
| 遥测数据 | `telemetry/` | 失败事件、错误日志 |

**安全机制：** 采集 Agent 在上传前自动移除 `.credentials.json`、`session-env/`、`cache/` 等敏感目录和文件。传输全程 HTTPS 加密。原始数据包服务端保留 30 天后自动清理。

## 35+ 评分维度详解

### 总览

5 大类、35+ 个具体维度。每个维度 0-100 分，按类别加权汇总为总分。

```
总分 = 活跃度(20%) + 使用质量(25%) + AI认知(15%) + 效率指标(25%) + 资源合理性(15%)

等级 = S(90+) / A(80-89) / B(70-79) / C(60-69) / D(<60)
```

---

### 1. 使用活跃度（权重 20%）—— 你到底用没用？

> 最基础的维度。不用 AI 的人，连评价的资格都没有。

| 维度 | 数据来源 | 评分逻辑 | 为什么重要 |
|------|----------|---------|-----------|
| 月活跃天数 | `history.jsonl` 时间戳 | 活跃天/工作日，100% = 满分 | 最直接的使用指标，每天都用才是基本功 |
| 日均会话数 | `sessions/` 文件数 | 合理区间（2-8次/天）得高分 | 太少说明没在用，太多可能在刷 |
| 日均对话轮数 | `sessions/` 轮数统计 | 反映每次交互的深度 | 深度对话说明在解决真实问题 |
| 会话时长分布 | `sessions/` 时间跨度 | 有效工作时段占比 | 区分工作使用和无效闲聊 |
| 连续不使用天数 | `history.jsonl` 时间间隔 | 超过 3 天无使用标记警告 | 长期不用需要关注原因 |
| 非工时使用统计 | `history.jsonl` 时间戳 | 仅记录，不作为扣分项 | 加班卷的人会加分（开玩笑，仅统计） |

---

### 2. 使用质量（权重 25%）—— 用了，但用得好吗？

> 打开 Claude 说一句"你好"然后关掉，算用了吗？当然不算。

| 维度 | 数据来源 | 评分逻辑 | 为什么重要 |
|------|----------|---------|-----------|
| 项目覆盖度 | `history.jsonl` project 字段 | 在多少个不同项目中使用了 AI | 只在一个项目用说明没养成习惯 |
| 平均有效轮数 | `sessions/` 排除 ≤2 轮会话 | 排除打开就关的无效会话 | 有效对话才产生价值 |
| 工具使用多样性 | `sessions/` tool_calls | 是否用到 Read/Edit/Grep/Agent/Bash 等多种工具 | 只会让 AI 聊天的不算真正会用 |
| 复杂任务占比 | `sessions/` 轮数 × 工具种类 | 长会话 + 多工具 = 复杂任务 | 复杂任务才是 AI 真正发挥价值的场景 |
| 上下文压缩频率 | `sessions/` compaction 记录 | 高频压缩说明会话内容丰富 | 长而深的会话更可能在做有意义的工作 |
| 任务完成率 | `tasks/` 完成/创建比 | 创建了多少任务，完成了多少 | 有始有终体现工作质量 |
| 规划能力使用率 | `plans/` 文件数 | 是否使用 plan 功能做复杂规划 | 会做计划的人效率更高 |

---

### 3. AI 认知与配置（权重 15%）—— 你懂不懂 AI？

> 不会调参数、不会配规则的人，用 AI 就像开法拉利送外卖。

| 维度 | 数据来源 | 评分逻辑 | 为什么重要 |
|------|----------|---------|-----------|
| 模型切换合理性 | `history.jsonl` /model 命令 | 是否根据任务选择合适模型（简单用 Haiku，复杂用 Opus） | 理解模型差异说明有 AI 素养 |
| 规则配置完善度 | `CLAUDE.md` + `rules/` | 是否建立了个性化 AI 工作规范 | 有规则的人输出质量稳定可控 |
| Memory 使用情况 | `projects/*/memory/` | 是否主动积累项目知识 | 善用记忆的人 AI 越用越聪明 |
| Settings 自定义度 | `settings.json` 配置项数 | 权限、环境变量等高级配置 | 深度定制说明真正理解工具 |
| Skills 使用频率 | `history.jsonl` 斜杠命令 | 是否探索和使用扩展能力 | 不探索新功能的人永远停在初级水平 |
| Hook 自动化配置 | `settings.json` hooks 字段 | 是否配置了自动化工作流 | 配 Hook 的人效率是不配的人的 N 倍 |

---

### 4. 效率指标（权重 25%）—— 用 AI 解决问题的速度

> 同一个 bug，A 用 3 轮对话解决，B 开了 5 个新会话还没解决。谁强谁弱？

| 维度 | 数据来源 | 评分逻辑 | 为什么重要 |
|------|----------|---------|-----------|
| 问题首次解决率 | `sessions/` 模式分析 | 同一问题是否反复开新会话 | 一次解决说明提问质量高 |
| 平均任务完成轮数 | `sessions/` + `tasks/` | 越少轮次完成任务越高效 | 高手用更少对话完成更多事 |
| 错误恢复速度 | `sessions/` 错误→修复间隔 | 出错后多快恢复正常工作流 | 遇到错误不慌的人效率更高 |
| 会话放弃率 | `sessions/` 分析 | 开了但没产出就结束的会话比例 | 高放弃率说明不会有效提问 |
| Git 产出关联 | `sessions/` 中 git 命令 | 会话是否转化为实际代码提交 | 最终看产出，不看过程 |
| 重复问题比例 | `history.jsonl` 相似度分析 | 反复问同类问题说明没有积累 | 问一次学一次才是正确的用法 |

---

### 5. 资源使用合理性（权重 15%）—— 有没有在浪费？

> Token 不是免费的。用得多不等于用得好。

| 维度 | 数据来源 | 评分逻辑 | 为什么重要 |
|------|----------|---------|-----------|
| Token 消耗趋势 | `telemetry/` + 会话大小 | 月度趋势分析，标记异常飙升 | 突然暴增可能在刷量 |
| 空转会话比例 | `sessions/` 分析 | 大量 token 但无实际产出 | 最典型的浪费行为 |
| 大文件重复读取 | `sessions/` 工具调用 | 同一个文件反复无截断读取 | 不会用搜索工具就是在烧钱 |
| 重复操作比例 | `sessions/` 分析 | 同一文件反复 read/edit | 低效的标志 |
| 无效命令比例 | `history.jsonl` | 错误命令、被拒绝的操作占比 | 高错误率说明操作不熟练 |

## 评分公式

```
单维度得分 = f(原始指标) → 0~100 分（每个维度有独立的评分函数）

类别得分 = 该类别下所有维度的平均分

总分 = 活跃度均分 × 20%
     + 使用质量均分 × 25%
     + AI认知均分 × 15%
     + 效率指标均分 × 25%
     + 资源合理性均分 × 15%

等级划分：
  S 级 (90+)  —— AI 原住民，标杆员工
  A 级 (80-89) —— 熟练使用者，值得表扬
  B 级 (70-79) —— 基本合格，还有提升空间
  C 级 (60-69) —— 需要改进，建议培训
  D 级 (<60)   —— 几乎没在用，需要谈话
```

## Web 管理后台

### 5 大核心页面

**1. 总览仪表盘** `/dashboard`
- 全公司平均分、等级分布饼图（S/A/B/C/D 各多少人）
- 本月 vs 上月趋势对比
- Top 10 / Bottom 10 员工排行
- AI 使用率热力图（日期 × 员工矩阵）

**2. 员工列表** `/employees`
- 表格展示：姓名、部门、总分、等级、趋势箭头（↑↓→）
- 支持按部门筛选、按各类分数排序
- 搜索框快速定位，点击进入详情

**3. 员工详情** `/employees/:id`
- 五维雷达图（活跃度/质量/认知/效率/资源）
- 各维度展开的明细分数条形图
- 最近 6 个月趋势折线图
- 最近上传记录和数据健康状态

**4. 月度报告** `/reports`
- 按月份选择，展示全公司评分报告
- 部门对比柱状图
- 等级变动明细（谁升了谁降了）
- 支持导出 PDF

**5. 系统管理** `/settings`
- 员工管理（增删改、生成采集 Agent API Key）
- 维度权重配置展示（后续支持自定义）
- 上传日志和异常监控

## 快速开始

### Docker 部署（推荐）

```bash
git clone https://github.com/zzhou23/colleague-kpi.git
cd colleague-kpi
cp .env.example .env  # 编辑配置
docker compose up -d
```

访问 `http://localhost` 即可看到管理后台。

### 部署采集 Agent

在每台员工机器上：

```bash
pip install ai-perf-review-agent
# 配置环境变量
export APR_AGENT_SERVER_URL=https://your-server.com
export APR_AGENT_API_KEY=员工专属key

# 启动定时采集（每日凌晨自动上传）
python -m agent

# 或手动运行一次
python -m agent --once
```

## FAQ

**Q: 这不就是监控员工吗？**
A: 这是绩效考核工具。就像你考核代码提交量、Bug 修复率一样，AI 使用效率也应该纳入考核体系。公司花钱买了 AI 工具，员工有义务有效使用。

**Q: 员工会不会反感？**
A: 如果公司都在用 AI 了但你还没用起来，你应该担心的是被淘汰而不是被监控。这个工具恰恰能帮你发现自己的 AI 使用短板，针对性提升。

**Q: 有人故意刷量怎么办？**
A: 35+ 个维度交叉验证。刷 token 数但质量维度不达标？空转会话比例高？重复操作多？所有这些都会被识别。骗得了一个维度，骗不了所有维度。

**Q: 会采集代码内容吗？**
A: 不会。我们只采集 Claude Code 的元数据（命令历史、会话统计、配置信息等），不采集任何实际代码内容或对话详情。所有敏感文件在上传前自动脱敏。

**Q: 只支持 Claude Code 吗？**
A: 目前以 Claude Code 为首个支持对象（因为它的本地数据最丰富、最适合分析）。后续会扩展到 GitHub Copilot、Cursor 等主流 AI 编程工具。
